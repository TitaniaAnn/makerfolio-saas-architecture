# 10 — Custom domains & on-demand TLS

**Audience:** developers working on makerfolio-saas.
**Scope:** how a Pro tenant connects their own hostname, how DNS ownership is proven, and how Caddy issues a Let's Encrypt cert on demand — gated so we never burn the LE rate limit.

> Pro tenants point their own domain straight at the box (bypassing the Cloudflare that fronts the platform domain). When a visitor first hits that hostname, Caddy issues a real Let's Encrypt cert on demand — but only after asking the internal `/caddy-ask` endpoint, which says yes only for domains the tenant has DNS-verified. The whole subsystem exists to make that one gate safe (CLAUDE.md invariant 7).

## The flow
**1. Add.** Owner of a Pro tenant opens `/admin/domains/` — the page is gated by `enforce_plan_flag('allow_custom_domain', …)` and `Plan::canCreate('custom_domains', …)` ([`index.php:5,19`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/index.php)). `add.php` POSTs a hostname through `TenantDomain::create()` ([`add.php:21`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/add.php)), which validates RFC-1035 shape ([`TenantDomain.php:54`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), rejects a hostname already claimed or under our own `PLATFORM_DOMAIN` ([`TenantDomain.php:131,136`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), stamps `is_apex` from the PSL ([`:146`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), and inserts a `PENDING_DNS` row with a 32-hex `verification_token`. If the hostname is an apex it auto-adds a `www.` sibling row (`auto_added_www_sibling=1`) that verifies/provisions independently ([`:159`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)).

**2. DNS setup.** `show.php` renders the two records the tenant must add at their registrar: a **CNAME** → `DomainVerifier::cnameTarget()` (`tenants.<PLATFORM_DOMAIN>`) and a **TXT** at `_makerfolio-verify.<hostname>` carrying the token ([`show.php:17,18,65-73`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/show.php)). Apex domains get an ALIAS/ANAME note since apexes can't take a CNAME ([`show.php:58`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/show.php)).

**3. Verify.** Either the 5-minute `verify_pending_domains` cron (ARCHITECTURE.md §6) or the tenant's "Verify now" button hits `verify.php`. It runs `DomainVerifier::verifyCname()` then — only if that passed — `verifyTxt()` ([`verify.php:37,41`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/verify.php)). On success it calls `TenantDomain::transitionTo($id, 'DNS_VERIFIED', …)` ([`verify.php:47`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/verify.php)); on failure it records `last_verification_error` + bumps the attempt count without a status change ([`verify.php:56`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/domains/verify.php)). DoH (not `dns_get_record`) is used to dodge the container resolver's stale cache and to get exact match flags ([`DomainVerifier.php:4-9`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/DomainVerifier.php)).

**4. Cert on demand.** A visitor hits `https://<hostname>` — or, more likely, the 5-minute `monitor-provisioning-domains` cron does: it TLS-probes every `DNS_VERIFIED`/`CERT_PROVISIONING` row with SNI ([`TlsCertProbe.php`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TlsCertProbe.php)), so provisioning starts without waiting for a first visitor. Caddy has no cert yet, so its `on_demand_tls` block GETs `http://localhost:8080/caddy-ask?domain=<hostname>` ([`Caddyfile.prod:27`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)). `caddy-ask/index.php` returns **200 only** when the row exists and its status is in `['DNS_VERIFIED','CERT_PROVISIONING','ACTIVE']` ([`caddy-ask/index.php:63`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/caddy-ask/index.php)); everything else (absent, `PENDING_DNS`, `FAILED_*`, `DISABLED`, invalid shape) is a 404 (`not-allowed`). An allowed ask for a `DNS_VERIFIED` row also transitions it to `CERT_PROVISIONING` — issuance is starting, which starts the failure-timeout clock. On 200 Caddy completes the ACME challenge and gets the LE cert.

**4b. Activate.** The next cron tick's probe sees a live cert covering the hostname and transitions the row to `ACTIVE`, stamping `cert_issued_at` + `cert_expires_at`. A row stuck in `CERT_PROVISIONING` >15 minutes without a served cert goes to `FAILED_CHALLENGE`; a probe that can't even reach the server is treated as inconclusive (logged, no transition) so a network blip can't mass-fail domains. Probes connect to the hostname itself by default (end-to-end — it also proves the tenant's DNS still points at us); set `CADDY_TLS_PROBE_CONNECT=caddy:443` if the box can't hairpin to its own public IP.

**5. Route.** Now TLS-terminated, the request reaches PHP-FPM. Bootstrap's `TenantResolver` looks up `tenant_domains` by exact hostname **and** `status = 'ACTIVE'` ([`TenantResolver.php:293-300`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantResolver.php)) — `DNS_VERIFIED`/`CERT_PROVISIONING` deliberately do **not** route (serving plaintext pre-cert defeats the point), they 404 via the apex-router. From there the request is identical to a subdomain tenant (`setSchema`, then the inherited controllers run unmodified).

`transitionTo` is the only writer of `status`. It validates against `ALLOWED_TRANSITIONS` ([`TenantDomain.php:38`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), writes whitelisted context columns, rotates the token on any `* → PENDING_DNS` re-verify hop ([`:226`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), audits via `PlatformActivity`, flips `TenantResolver::invalidateHost()` when crossing the ACTIVE boundary ([`:248`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)), and fires a (failure-swallowed) tenant email on `DNS_VERIFIED`/`ACTIVE` — skipping the auto-www sibling so the pair sends one message ([`:257`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantDomain.php)).

## The Caddyfile wiring
Three pieces in [`Caddyfile.prod`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod):

- **Global `on_demand_tls { ask … }`** ([`:26-28`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)) — the single gate. `ask` alone; see the gotcha below about the removed `interval`/`burst`.
- **`makerfolio.art, *.makerfolio.art`** ([`:31`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)) — the platform block, served with a static Cloudflare Origin cert (`tls /etc/caddy/certs/origin.crt …`). This is NOT on-demand: Cloudflare fronts it and the origin cert's SANs already cover apex + every subdomain.
- **Catch-all `https:// { tls { on_demand } }`** ([`:78-81`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)) — matches any hostname that *isn't* the platform block (more-specific site wins), i.e. exactly the custom domains. Its routing is a byte-for-byte copy of the platform block's `route`/`handle_errors`, kept inline (not refactored) so the platform block can't regress.
- **Internal `http://localhost:8080`** ([`:111`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)) — serves `/caddy-ask` over plain HTTP to PHP-FPM. It's `localhost:8080` **inside the Caddy container**: not published in compose and `ufw` opens only 22/80/443, so only Caddy itself can reach the ask endpoint. Note it's `http://localhost:8080`, distinct from `php_fastcgi web:9000` — `web:9000` is the **FastCGI** port (the CGI protocol), not an HTTP server, so it can't be the `ask` target; the ask must be an HTTP URL, hence the dedicated `:8080` HTTP site that itself proxies to `web:9000` via FastCGI.

## Invariants & gotchas
- **Ask returns 200 ONLY for verified domains** — the LE rate-limit guard (CLAUDE.md invariant 7). Returning 200 for arbitrary hostnames lets an attacker point 10K domains at our IP and exhaust the 50-certs/registered-domain/week limit; the 404 is the only thing standing in the way ([`caddy-ask/index.php:9-13`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/caddy-ask/index.php)). The only path into `DNS_VERIFIED` is completing the TXT ownership challenge, which an attacker can't forge for a domain they don't control (ARCHITECTURE.md §Custom-domain TLS abuse).
- **Tenant-status cert cutoff (Q1)** — even a DNS-verified domain is refused (404) if its tenant is `PENDING_DELETION`/`DELETED`, or `SUSPENDED` for ≥30 days ([`caddy-ask/index.php:79-98`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/caddy-ask/index.php)). The first 30 days of suspension keep the cert so the friendly suspended page still serves over HTTPS.
- **Don't trust the resolver before the cert** — `TenantResolver` routes `status = 'ACTIVE'` only ([`TenantResolver.php:298`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/TenantResolver.php)); the ask gate is broader (`DNS_VERIFIED`/`CERT_PROVISIONING`/`ACTIVE`) because the cert must exist *before* the row can reach ACTIVE. Keep these two allowlists distinct — they are not the same set on purpose.
- **PSL prevents registrable-domain abuse / drives UX** — `isApex` decides auto-www pairing and apex-vs-subdomain DNS guidance; a missing PSL data file throws rather than silently misclassifying ([`PublicSuffixList.php:130`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/PublicSuffixList.php)). Refresh via `bin/refresh-psl.php`.
- **Caddyfile change = restart, not reload** — `admin off` is set ([`Caddyfile.prod:17`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/Caddyfile.prod)), so there's no admin API to hot-reload config; a change requires a container restart. This is also why the domain crons observe certs via TLS probe rather than the admin API the original Phase 4 plan called for: with `admin off` there is nothing to poll, and the earlier admin-API version of `monitor-provisioning-domains` silently skipped in production (`CADDY_ADMIN_URL` unset → exit 0), leaving no code path that could ever transition a domain to `ACTIVE`.
- **`caddy validate` ≠ `caddy run` (the outage lesson)** — PR #133 shipped `on_demand_tls { ask … interval … burst … }`. `caddy validate` accepted `interval`/`burst`, but newer Caddy removed them and `caddy run` *rejected* them on restart, crash-looping Caddy. Fix commit `8674342` dropped both. **Validate with `caddy adapt`, not just `caddy validate`, before restarting.**
