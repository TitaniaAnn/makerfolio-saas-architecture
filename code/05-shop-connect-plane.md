# 05 — The shop: the Stripe Connect plane

**Audience:** developers working on makerfolio-saas.
**Scope:** per-tenant storefront payments — checkout, fulfillment, and Connect-account onboarding/state on each tenant's OWN Stripe account.

> Every tenant's shop takes card payments into the **tenant's own** Stripe account (BYO-Connect), not the platform's. The platform Stripe account creates the connected accounts (`Account::create`) and creates Checkout Sessions *on* them via the `stripe_account` request option (direct charges), so the maker is merchant of record and the money lands in their balance. This is the **shop** plane — distinct from the platform-subscription plane (walkthrough `04`); CLAUDE.md invariant 3 says the two never cross. Design: [STRIPE_CONNECT_INTEGRATION.md](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/design-docs/STRIPE_CONNECT_INTEGRATION.md) (Phase 9, PRs #94–#97).

## The flow

The state machine the whole plane turns on lives in `ShopConnect::TRANSITIONS` ([ShopConnect.php:32](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L32)):

```
NONE ─▶ ONBOARDING ─▶ ENABLED ⇄ RESTRICTED ─▶ DISABLED ─▶ ONBOARDING
```

`stripe_connect_status` is the human-facing summary; `stripe_connect_charges_enabled` is the flag the shop actually reads.

1. **Onboard (UX16 wizard).** A maker hits `/admin/shop/` with `allow_shop` on but not yet ENABLED; the hero card ([index.php:70](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/index.php#L70)) offers two CTAs — "connect a Stripe account I have" (Standard) and "set up a new account" (Express) — each POSTing to [connect.php](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/connect.php). `ShopConnect::createAccount` ([ShopConnect.php:268](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L268)) calls `\Stripe\Account::create(['type' => 'standard'|'express', 'metadata' => ['tenant_id' => ...]])` **on the platform key** ([ShopConnect.php:282](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L282)), stamps `stripe_connect_account_id` + `stripe_connect_account_type`, and transitions `NONE → ONBOARDING`. It is idempotent — an existing account short-circuits before any Stripe call ([ShopConnect.php:274](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L274)). `onboardingLink` ([ShopConnect.php:301](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L301)) mints an `AccountLink` (`type: account_onboarding`) and the controller redirects to Stripe-hosted KYC ([connect.php:30-35](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/connect.php#L30)).

2. **Return + reconcile.** Stripe redirects back to [connect-return.php:15](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/connect-return.php#L15), which calls `refreshFromStripe` → `\Stripe\Account::retrieve` → reads `charges_enabled` / `details_submitted` / `requirements.currently_due` → `applyAccountFlags` → `deriveStatus` → ENABLED (charges on) or RESTRICTED (details in, charges still off) and flashes the maker accordingly. The maker may not finish in this round-trip — step 5 covers the async case.

3. **Catalog.** `add-product.php` is the inherited product controller, untouched except a soft "Stripe isn't payment-ready" nudge when `uiStateForTenant()['ready']` is false ([add-product.php:235](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/add-product.php#L235)) — so a maker doesn't build a catalog that can't sell.

4. **Checkout.** Public `/shop` renders the Buy button only when `ShopConnect::shopPaymentReady($tid)` is true ([shop.php:16](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop.php#L16)). The POST hits [checkout.php](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/checkout.php): plan gate (`allow_shop`, [checkout.php:14](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/checkout.php#L14)) → `shopPaymentReady` gate ([checkout.php:26](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/checkout.php#L26)) → CSRF → `StripeHelper::createCheckoutSession`. There [Stripe.php:110](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L110) reads `ShopConnect::checkoutRouting()`; if non-null it swaps the API key to `STRIPE_PLATFORM_SECRET_KEY` and creates the session with the `['stripe_account' => acct_xxx]` request option ([Stripe.php:118](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L118)) — a **direct charge** that settles into the maker's balance — optionally setting `payment_intent_data.application_fee_amount` (0 bps today, [Stripe.php:112](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L112)). Self-host with no routing falls through to the legacy single-key `Session::create($params)` ([Stripe.php:120](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L120)). Either way a `pending` row is inserted into `orders` ([Stripe.php:124](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L124)) keyed by `stripe_session_id`. The success/cancel/webhook return paths stay un-gated so an in-flight order from before a plan downgrade can still complete ([checkout.php:7-12](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/checkout.php#L7)).

5. **Fulfill.** `checkout.session.completed` arrives at the **Connect** webhook ([connect/index.php:46](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/platform-webhook/connect/index.php#L46)) → `ShopOrders::processConnectShopEvent` ([ShopOrders.php:98](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L98)): resolve `schema_name` from `public.tenants`, `setSchema`, **INSERT `stripe_webhook_events` first** ([ShopOrders.php:112](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L112)), then `Database::transaction` runs `markCheckoutPaid` (pending→paid + writes customer/shipping fields + atomic stock decrement, [ShopOrders.php:21](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L21)) and stamps `processed_at` ([ShopOrders.php:141](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L141)). **Mail dispatches AFTER the transaction** ([ShopOrders.php:150](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L150)); `resetSchema` runs in `finally` ([ShopOrders.php:164](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L164)). The order is now visible in [public/admin/orders/](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/orders/) — a Connect-agnostic inherited list/view/ship surface reading `orders` in the tenant schema.

6. **Async readiness.** `account.updated` ([connect/index.php:38](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/platform-webhook/connect/index.php#L38)) → `applyAccountFlags` keeps `charges_enabled`/status current even when the maker finishes KYC later, outside the return flow — the webhook, not the return redirect, is the source of truth for readiness.

**Refunds/disputes are not mediated** (Q5): with direct charges the maker is merchant of record and owns refunds in their own Stripe dashboard. The endpoint does **not** subscribe `charge.refunded`/`charge.dispute.*`, so a Stripe-side refund does not auto-flip the makerfolio order — the `orders.refunded` status is set manually by the maker. Auto-sync is a documented fast-follow.

## Two webhooks, two planes

| | Self-host shop webhook | SaaS Connect webhook |
| --- | --- | --- |
| File | [public/shop/webhook.php](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/webhook.php) | [public/platform-webhook/connect/index.php](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/platform-webhook/connect/index.php) |
| Routing | Host-resolved (tenant from Host header) | `/platform-webhook/*` — TenantResolver skips host resolution; tenant comes from `event.account` |
| Tenant resolution | implicit (`search_path` already set in bootstrap) | `ShopConnect::resolveTenantByAccount($event->account)` ([connect/index.php:34](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/platform-webhook/connect/index.php#L34)), reverse-lookup on the unique index on `public.tenants.stripe_connect_account_id` |
| Signing secret | `STRIPE_WEBHOOK_SECRET` ([Stripe.php:142](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L142)) | `STRIPE_CONNECT_WEBHOOK_SECRET` ([ShopConnect.php:404](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L404)) |
| Events | `checkout.session.completed`, `payment_intent.payment_failed` | + `account.updated` (Connect readiness) |
| Used by | self-host single-key deployments only | SaaS tenants (charges on connected accounts) |

Why the split: Connect events arrive keyed by `acct_xxx` with **no Host context**, so the host-based tenant resolver has nothing to key on — they must reverse-resolve the account id, hence a dedicated platform endpoint ([connect/index.php:1-9](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/platform-webhook/connect/index.php#L1)). Both secrets default to `''` in [config/config.php:210,276](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/config/config.php#L210); both honor a stub-bypass literal (`whsec_test_skip_verify`, [ShopConnect.php:400](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L400)) mirroring the billing webhook, so smokes run keyless.

Note the **third Stripe key**: connected accounts and their charges live on `STRIPE_PLATFORM_SECRET_KEY` ([ShopConnect.php:248](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L248), [Stripe.php:116](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L116)) — the *same* account that runs subscription billing (walkthrough `04`), with Connect enabled. The inherited `STRIPE_SECRET_KEY` ([Stripe.php:35](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L35)) is **self-host only** from Phase 9 on.

## Invariants & gotchas

- **Webhook idempotency contract (INSERT-first).** Claim the event by INSERTing into `<tenant>.stripe_webhook_events` *before* handling; the PK on `event_id` serializes concurrent retries. Duplicate-key with `processed_at IS NOT NULL` → real dup, ack and stop; `processed_at IS NULL` → a prior attempt crashed mid-handler, re-run. The handler + the `processed_at` stamp run together inside `Database::transaction`, so a crash leaves the row in the "claimed but unprocessed" state for Stripe's retry. Both webhooks implement this identically ([ShopOrders.php:110-145](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L110), [webhook.php:42-87](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/webhook.php#L42)). Break it and Stripe retries cause double stock decrements / double receipts. Two subtleties: the PK is the string `event_id`, so the Connect INSERT passes `null` for the serial-id arg ([ShopOrders.php:115](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L115)); and `markCheckoutPaid` is *also* order-level idempotent — only `pending` orders transition, so a re-delivered event for an already-paid/shipped/refunded order is a no-op ([ShopOrders.php:32](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L32)).

- **Atomic stock decrement.** `markCheckoutPaid` decrements `quantity` and flips `status` to `sold` in **one** UPDATE evaluated against the OLD row, closing the concurrent-checkout race ([ShopOrders.php:60-76](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L60)). It has a Postgres branch and a MySQL branch — the Postgres one derives the sold-out check from `quantity - ?` because Postgres evaluates every SET against the pre-update row.

- **Mail AFTER the transaction.** `Mailer::notifyOwnerOfSale` / `sendCustomerReceipt` dispatch outside the tx ([ShopOrders.php:150](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopOrders.php#L150), [webhook.php:98](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/shop/webhook.php#L98)), each wrapped in its own try/catch so a mail failure can't roll back a paid order or 500 the webhook. `mail()` can take seconds; running it inside the webhook risks blowing Stripe's 10s deadline and cascading retries. The tenant schema is still set, so the tenant's own `settings` brand the mail; From is the verified platform sender with Reply-To back to the maker (`Mailer::notifyOwnerOfSale` sets Reply-To to the *customer* so the maker can reply straight to the buyer, [Mailer.php:39](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Mailer.php#L39)) — full From/Reply-To treatment in `06`.

- **`charges_enabled` is load-bearing; status is cosmetic.** `deriveStatus` returns ENABLED iff `charges_enabled`, RESTRICTED if details/requirements are outstanding, else ONBOARDING ([ShopConnect.php:65-74](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L65)); `shopPaymentReady` = `allow_shop AND charges_enabled` ([ShopConnect.php:437](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L437)). The status string drives UI copy only — never gate a charge on it.

- **DISABLED is sticky.** `applyAccountFlags` never auto-reactivates a maker who explicitly disconnected — it pins DISABLED regardless of what Stripe reports ([ShopConnect.php:364](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L364)) so a late `account.updated` (charges back on) can't silently re-enable charges behind the maker's back. Re-enabling is an explicit reconnect: [connect.php:26](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/connect.php#L26) moves DISABLED→ONBOARDING *before* re-onboarding so the sticky guard doesn't strand them. `disconnect()` ([ShopConnect.php:408](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L408)) zeroes `charges_enabled` locally but leaves the Standard/Express account itself with the maker at Stripe.

- **State changes go through `transitionTo()`** ([ShopConnect.php:197](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L197)), validating against `TRANSITIONS`; self-transition is an idempotent no-op. Reconciliation (`applyAccountFlags`) writes the *derived* status directly (Stripe is the source of truth) rather than guarding through the user-transition machine — the one deliberate exception, still respecting the DISABLED pin.

- **Account id lives on `public.tenants` (invariant 2).** No FK crosses the schema boundary; the Connect webhook needs an `acct_xxx → tenant` reverse lookup *before* any `setSchema` (the event has no schema/host context), so it must be a public-side, unique-indexed column. `resolveTenantByAccount` is that O(1) lookup ([ShopConnect.php:379](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L379)).

- **SaaS vs self-host branch.** `isSaaSShopContext()` = `PLATFORM_DOMAIN` set AND `TenantResolver` exists ([ShopConnect.php:426](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L426)). Self-host falls back to `STRIPE_ENABLED`/`STRIPE_SECRET_KEY` everywhere; the two paths never coexist in one deployment, so there's no grandfather shim (Q3). `checkoutRouting()` returns null off-SaaS or when the tenant isn't charges-enabled — a defensive guard, since callers gate via `shopPaymentReady` first ([ShopConnect.php:460-481](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L460)).

- **Onboarding controllers are OWNER-gated + CSRF.** `connect`/`connect-return`/`disconnect` all `Auth::requireLogin()` + `require_role(Role::OWNER)` + `enforce_plan_flag('allow_shop')`; the POST ones `csrf_verify()` (e.g. [connect.php:6-14](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/connect.php#L6)). Onboarding is billing-adjacent, so EDITOR can manage the catalog but not the money path (the hero card shows EDITORs a note instead of the CTAs, [index.php:74](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/shop/index.php#L74)).

- **API version pinned.** The Connect plane pins `PlatformStripe::PINNED_API_VERSION` ([ShopConnect.php:249](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/ShopConnect.php#L249), [Stripe.php:117](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Stripe.php#L117)) so payload shapes stay stable across SDK upgrades — the webhook handlers read raw event fields by name.
