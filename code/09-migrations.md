# 09 — Migrations & schema management

**Audience:** developers working on makerfolio-saas.
**Scope:** how fresh-install SQL, incremental migrations, the runner, the per-tenant fan-out, and the install wizard fit together for Postgres schema-per-tenant.

> Fresh installs load a canonical schema file (`sql/init.postgres.sql` for tenant schemas, `sql/public/001_platform.sql`+ for `public`) that pre-seeds the `schema_migrations` ledger so the runner has nothing to replay. Incremental changes ship as numbered `NNN_*.sql` files: tenant migrations in [`sql/migrations.postgres/`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/migrations.postgres) tracked in each `tenant_<id>.schema_migrations`, public migrations in [`sql/public/`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/public) tracked in `public.schema_migrations`. Two ledgers, same table shape, same idempotency contract.

## The flow

**Fresh install.** `Installer::initializeSchema` ([Installer.php:187](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Installer.php#L187)) reads `init.postgres.sql` and runs the whole file in a single `$pdo->exec($sql)` — *not* through the splitter — so the `$$`-quoted `set_updated_at()` trigger body survives (the splitter would cut it at internal `;`-newlines). That file ends with an `INSERT … ON CONFLICT (version) DO NOTHING` that pre-marks every migration 001–028 as `source='mark'` ([init.postgres.sql:499](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/init.postgres.sql#L499)). Net effect: a fresh tenant schema already has the post-028 table shape *and* a full ledger, so the runner finds nothing pending. Public is the symmetric case — `sql/public/001_platform.sql` is single-statement-per-`;`, no `$$` blocks, so it goes through `applyMigrations` like any other public file ([001_platform.sql:9](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/public/001_platform.sql#L9)).

**Incremental.** `MigrationRunner::pending()` = files on disk matching `^\d+_[A-Za-z0-9_\-]+\.sql$` ([MigrationRunner.php:80](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L80)) minus ledger rows. `apply()` splits the file, `exec`s each statement, and on success records the version via `recordVersion` (`ON CONFLICT (version) DO NOTHING`, [MigrationRunner.php:175](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L175)). On a statement failure it throws **without** recording — DDL has already implicitly committed the earlier statements, so the admin fixes the cause and re-runs ([MigrationRunner.php:149](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L149)). `Installer::applyMigrations` ([Installer.php:221](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Installer.php#L221)) is a parallel, PDO-direct copy of the same loop used by the installer and Docker boot.

**Per-tenant fan-out.** `PlatformMigrationRunner::applyAll` ([PlatformMigrationRunner.php:41](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/PlatformMigrationRunner.php#L41)) runs public first (`applyToPublic` → `Installer::applyMigrations` at `sql/public/`), then `applyToActiveTenants`: for each `Tenant::active()` row it `setSchema($schema)`, news a `MigrationRunner` (which resolves to `sql/migrations.postgres/` on pgsql), and applies that tenant's pending set. Each iteration is wrapped in `try/catch` recording the error into a per-tenant map, with `Database::resetSchema()` in a `finally` ([PlatformMigrationRunner.php:72](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/PlatformMigrationRunner.php#L72)) — one bad tenant never blocks the rest and never leaks a `search_path`.

**Docker boot.** [`docker/auto-migrate.php`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/docker/auto-migrate.php) runs on container start. On the multi-tenant path it applies **only** `sql/public/` migrations and skips `bootstrapAuthFromEnv` (auth settings live inside each tenant schema, not public — [auto-migrate.php:62](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/docker/auto-migrate.php#L62)). Tenant-schema migrations are *not* applied at boot — that's the operator-triggered `PlatformMigrationRunner` job ([bin/run-migrations.php](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/bin/run-migrations.php) or the platform-admin B-page).

## Writing a migration (rules)

- **Pick the ledger.** Public-schema change → `sql/public/NNN_*.sql`. Per-tenant app table → `sql/migrations.postgres/NNN_*.sql`. They never mix; each has its own `schema_migrations` in its own schema.
- **One statement per line for the splitter.** [`splitStatements`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L259) strips `--`/`#` line comments and `/* … */` blocks, then splits on `;` followed by end-of-line: `preg_split('/;\s*(\R|$)/', $body)` ([MigrationRunner.php:279](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L279)). It does **not** understand string literals or PL/pgSQL bodies. So a `DO $$ … $$` block must be on a **single line** — every internal `;`-newline would otherwise cut it apart (see the header note + the one-line `DO` blocks in [001_pottery_images.sql:19](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/migrations.postgres/001_pottery_images.sql#L19)).
- **Idempotency is mandatory** (CLAUDE.md invariant 5). Guards in use:
  - `CREATE TABLE IF NOT EXISTS` for new tables ([001_platform.sql:18](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/public/001_platform.sql#L18)).
  - `DO $$ BEGIN IF NOT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_schema = current_schema() AND table_name = '…') THEN … END IF; END $$;` for conditional DDL ([001_pottery_images.sql:19](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/migrations.postgres/001_pottery_images.sql#L19)).
  - `INSERT … ON CONFLICT (…) DO NOTHING`/`DO UPDATE` for seed/upsert rows.
- **Pre-mark in `init.postgres.sql` when the DDL also lands in the canonical schema.** A new tenant migration must (a) ship as a `NNN_*.sql` file *and* (b) have its DDL folded into the init tables and its version added to the pre-mark `INSERT` ([init.postgres.sql:499](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/init.postgres.sql#L499)). Otherwise fresh installs would try to re-run DDL that init already created.

## Invariants & gotchas

- **Re-applies harmlessly if the ledger row is lost.** Every `NNN_*.sql` is guarded, so re-running its DDL is a no-op; only the *recording* INSERT could collide, and `recordVersion` swallows that with `ON CONFLICT DO NOTHING` ([MigrationRunner.php:176](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/MigrationRunner.php#L176)). `pg-smoke` proves this by clearing the ledger and replaying twice.
- **One bad tenant must not block others.** The per-tenant `try/catch/finally` in `applyToActiveTenants` records the error and continues; `finally` resets `search_path` even on failure ([PlatformMigrationRunner.php:79](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/PlatformMigrationRunner.php#L79)).
- **No FK crosses the schema boundary.** Public tables reference tenants only by `tenant_id`; tenant tables never FK to public ([001_platform.sql:6](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/public/001_platform.sql#L6), [009_tenant_domains.sql:9](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/sql/public/009_tenant_domains.sql#L9)). This keeps each tenant schema independently dump/drop/restore-able.
- **`init.*` is for empty DBs only.** `initializeSchema` uses `CREATE TABLE IF NOT EXISTS`, so re-running it against a populated schema silently skips tables — it will **not** migrate a stale schema forward ([Installer.php:185](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/includes/Installer.php#L185)). Use the runner for that.
- **Partial-failure state stays committed.** Postgres DDL auto-commits per statement; a mid-file failure leaves earlier statements applied and the ledger row absent. Re-run is safe *because* of the guards — but verify, don't assume.
- **The admin UI is self-host-only.** `public/admin/migrations/{index,run}.php` bail with a flash when `PLATFORM_DOMAIN` is set ([index.php:10](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/public/admin/migrations/index.php#L10)) — hosted tenants have no DB access and are migrated centrally.

## Tests & verification

- [`bin/pg-smoke.php`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/bin/pg-smoke.php) — against a live PG in a throwaway `pgsmoke_<rand>` schema: `initializeSchema` loads init (≥23 tables), reloads it (idempotent), then **clears the ledger and applies all migrations twice** ([pg-smoke.php:75](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/bin/pg-smoke.php#L75)) — proving the migration *SQL itself* (not just the ledger guard) is valid and replayable — and finally confirms a full ledger yields 0 pending.
- [`bin/platform-migrations-smoke.php`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/bin/platform-migrations-smoke.php) — proves the two `PlatformMigrationRunner` properties: happy-path `applyAll` across real tenants is clean, and a **phantom tenant** (a `public.tenants` row pointing at a non-existent schema) errors *in isolation* while real tenants stay clean and `search_path` is restored ([platform-migrations-smoke.php:58](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/bin/platform-migrations-smoke.php#L58)).
- [`tests/MigrationRunnerTest.php`](https://github.com/TitaniaAnn/makerfolio-saas/blob/main/tests/MigrationRunnerTest.php) — pure unit coverage of `splitStatements` (comments, multi-line `CREATE`, trailing-no-`;`) and `available()` discovery (excludes `init.sql`, non-numbered files); plus `test_real_migrations_parse_through_splitter` asserts every shipped `sql/` file splits to ≥1 statement. DB-touching paths are not unit-tested — the smokes cover those.
